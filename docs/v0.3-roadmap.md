# Super CDN v0.3 Roadmap

Status: next-session execution plan.

`v0.2.0` closed the preferred Web hosting split model: Cloudflare owns entry HTML and SPA fallback, while non-entry resources are delivered by explicit resource libraries such as AList/OpenList or IPFS/Pinata. `v0.3` should make that multi-resource foundation operationally reliable.

## Product Theme

Turn Super CDN from "can upload and host" into "can schedule, repair and explain delivery across resource libraries".

The primary product surface for `v0.3` is multi-resource-library governance:

- resource capability discovery
- replica state tracking
- primary plus backup replication
- explicit smart routing
- explicit static-resource failover
- route explanation and probes

Do not expand R2-backed Web hosting as a mainstream path. R2 remains the CDN/object acceleration line and legacy Web compatibility surface. Preferred Web hosting remains Cloudflare entry plus resources on AList/OpenList, Cloudflare-native static assets or IPFS/Pinata.

## Boundary Rules

- Go entry delivery remains for tests, integration and compatibility.
- Cloudflare entry mode remains the preferred production Web hosting shape.
- Static-resource failover is off by default and must be explicitly enabled.
- Static-resource failover never falls back to the Go origin.
- Homepage fallback to Go is only an explicit temporary operator action with warning headers and migration guidance.
- Smart routing and failover require at least two ready resource libraries.
- A degraded active manifest with one healthy resource is allowed only as recovery after refresh, not as a new smart-routing success state.

## Milestone 1: Resource Capability Model

Define a capability model for every resource library and expose it through diagnostics.

Progress:

- Started in `v0.3-dev`: `resource-status` now returns typed capability fields for configured resource libraries and direct resource-capable storage targets such as `ipfs_pinata`.

Required capabilities:

- can upload
- can delete remote objects
- can produce a public locator
- supports Range GET
- supports HEAD
- HTTP-only locator risk
- Web resource suitability
- CDN bucket suitability
- immutable CID behavior
- preferred cache policy

Expected implementation:

- Add typed capability fields to storage/resource-library models.
- Have R2, AList/OpenList, IPFS/Pinata and Cloudflare-native resources report capabilities consistently.
- Extend `resource-status` to show capability, health and last failure together.
- Keep credentials redacted.

Exit criteria:

- `supercdnctl resource-status` can explain what each library can safely do.
- Docs explain which libraries are valid for Web resources, CDN buckets and backups.

## Milestone 2: Replica State And Repair

Track object replicas as first-class state instead of treating upload success as the whole story.

Required states:

- `pending`
- `ready`
- `failed`
- `stale`
- `deleted`

Expected implementation:

- Persist per-object replica status, target, locator, CID or provider id where available.
- Record verification status after upload, not just provider response success.
- Add a repair path for missing or failed replicas.
- Add a refresh path for signed locators and provider metadata.

Candidate CLI:

```powershell
supercdnctl replicas -object-id 123
supercdnctl repair-replicas -object-id 123
supercdnctl refresh-replicas -bucket demo
```

Exit criteria:

- A failed backup upload is visible and actionable.
- A repaired replica can re-enter manifest candidate selection.

## Milestone 3: Replication Policy

Make primary plus backup replication an explicit policy rather than an incidental side effect.

Expected implementation:

- Route profiles declare primary and backups clearly.
- Upload/deploy paths replicate non-entry resources according to policy.
- Replication results are stored per file/object.
- Deployment should fail or warn based on policy strictness.

Policy modes:

- `primary_only`: upload to primary only.
- `best_effort_backups`: deploy when primary is ready, record backup failures.
- `require_backups`: require configured backups before activation.

Exit criteria:

- Hybrid Web deployments that request smart routing/failover cannot publish with only one ready source.
- CDN buckets can choose whether backup replication is required.

## Milestone 4: Smart Routing Formalization

Turn the current canary-level smart routing into a stable operator workflow.

Expected behavior:

- Route selection can use region, priority, weight and health.
- Candidate generation excludes recently unhealthy targets.
- Worker responses include route policy, selected target and reason headers.
- CLI can explain why a route chose a target.

Candidate CLI:

```powershell
supercdnctl route-explain -site cyberstream -path /assets/app.js
supercdnctl routing-policy-status -policy global_smart
```

Exit criteria:

- A user can see all candidates, selected candidate, rejected candidates and rejection reasons.
- Live smart-routing canary passes with at least two resource libraries.

## Milestone 5: Explicit Failover Hardening

Keep failover opt-in and make failure behavior boring.

Expected implementation:

- `deploy-site -resource-failover` remains the explicit switch.
- Worker failover tries ordered ready candidates only.
- If all candidates fail, Worker returns an edge error with route diagnostics.
- It never requests Go origin for static assets.
- Probe output distinguishes normal smart routing from explicit failover.

Exit criteria:

- Tests prove static-resource failure does not call origin.
- Live canary demonstrates primary failure and backup success.

## Milestone 6: Probes, Browser Checks And Speed Tests

Move the current manual verification habits into commands.

Expected implementation:

- Extend `probe-site` with browser white-screen detection.
- Record `#root` mount, console errors, mixed-content errors and screenshot path.
- Add direct asset speed tests for CDN bucket, AList/OpenList `/d`, IPFS gateway and Worker-proxied storage.
- Include Range GET and MIME/cache checks in one report.

Candidate CLI:

```powershell
supercdnctl probe-site -url https://demo.qwk.ccwu.cc/ -browser
supercdnctl speed-test -url https://demo.qwk.ccwu.cc/assets/app.js -proxy http://127.0.0.1:10808
supercdnctl speed-test-bucket -bucket demo -object docs/readme.md
```

Exit criteria:

- Browser render failures are caught without a manual Playwright script.
- Speed reports show first hop, final source, average throughput and cache/source headers.

## Milestone 7: Production Canary Matrix

Run a small but representative matrix before cutting `v0.3.0`.

Required canaries:

- Web: Cloudflare entry plus AList/OpenList primary plus IPFS backup.
- Web: Cloudflare entry plus IPFS primary.
- CDN bucket: R2 overseas object acceleration.
- CDN bucket: AList/OpenList domestic all-line acceleration.
- Failover: primary resource failure with backup success.
- Probe: browser render plus JS/CSS manifest asset checks.

Exit criteria:

- Every canary has a URL, deployment or bucket id, probe output and recorded source headers.
- No test relies on Go origin for static-resource recovery.

## Suggested Execution Order

1. Add the resource capability model.
2. Add replica state persistence and status output.
3. Add repair/refresh commands.
4. Formalize replication policy.
5. Add route explanation.
6. Harden explicit failover tests and Worker headers.
7. Automate browser/speed probes.
8. Run the production canary matrix.
9. Update docs and release `v0.3.0`.

## Non-Goals For v0.3

- Public SaaS UI.
- A full management dashboard.
- Self-hosted IPFS cluster operations.
- Automatic legal/content moderation workflows.
- Making R2 the mainstream Web resource target.

UI work should wait until the scheduling and replica core is stable. A UI built before the delivery model settles will churn heavily.

## Definition Of Done

- `go test ./...` passes.
- Worker `npm test` and `npx tsc --noEmit` pass.
- Capability, replica and routing docs are updated.
- At least one smart-routing Web canary uses two ready resource libraries.
- At least one explicit failover canary proves no Go-origin fallback for static resources.
- Browser probe catches render success/failure automatically.
- Release notes document the canary matrix and remaining boundaries.
