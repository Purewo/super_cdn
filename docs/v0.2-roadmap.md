# Super CDN v0.2 Roadmap

Status: next-version planning.

`v0.2.0` is the next feature line after the `v0.1.x` maintenance freeze. The headline feature is IPFS as the third major acceleration/storage surface, alongside Cloudflare-native overseas delivery and AList/OpenList domestic delivery.

## Product Theme

Bring IPFS from a low-level storage adapter into a first-class Super CDN capability.

Web hosting boundary for this roadmap:

- Go entry delivery is kept for simple tests, integration and compatibility.
- Preferred Web hosting puts the homepage/SPAs on Cloudflare Static/Pages and routes non-entry resources to AList/OpenList, Cloudflare-native static assets, or IPFS/Pinata.
- R2 stays maintained as the CDN/object acceleration line. R2-backed website flows are legacy compatibility and should not receive new mainstream Web hosting features.
- Smart routing and failover require at least two ready resource libraries and remain explicit opt-in.
- Static-resource failover never falls back to Go. Homepage fallback to Go is allowed only as an explicit last-resort operator action with a migration warning.

Current state:

- The codebase already has a Pinata/IPFS storage adapter.
- IPFS can exist as a backup storage target through the generic storage layer.
- It is not yet a first-class product workflow with dedicated health checks, gateway policy, CID visibility, pin lifecycle or CLI UX.

Target state:

- IPFS can be selected intentionally for durable decentralized asset storage.
- Super CDN can expose IPFS CIDs, gateway URLs and pin status in a predictable way.
- IPFS can participate in edge manifests and CDN bucket responses without making the Go service stream content.
- Operators can validate IPFS reachability before trusting it in a live route.

## Version Boundary

`v0.1.x` remains feature-frozen. IPFS work starts in `v0.2.0` or a dedicated feature branch.

Do not backport IPFS product changes into `v0.1.x` unless they are critical bug fixes for existing Pinata backup behavior.

## Milestone Checklist

### 1. Architecture Definition

- Define the IPFS role clearly: primary storage, backup storage, archive tier, public gateway tier, or route-policy target.
- Decide the first supported provider surface: Pinata first, raw IPFS node later.
- Define object identity fields: CID, gateway URL, pin id/provider id, content type, size, created time and pin status.
- Define whether IPFS objects are immutable-only in Super CDN v0.2.
- Define how IPFS fits into edge manifests: direct gateway redirect, gateway fallback list, or proxy fallback.

Exit criteria:

- `docs/v0.2-roadmap.md` has the accepted delivery model.
- Existing v0.1 delivery paths are not changed by default.

### 2. Provider And Config Hardening

- Formalize Pinata/IPFS config examples around Pinata v3 files APIs (`api_base_url` for list/delete/groups, `upload_base_url` for uploads).
- Add provider status checks similar to Cloudflare status where practical.
- Verify token validity without uploading data.
- Verify gateway base URL reachability.
- Make proxy behavior explicit and documented.
- Ensure secrets never appear in status responses or logs.

Exit criteria:

- `supercdnctl` can report IPFS/Pinata readiness without a live upload.
- Failed provider configuration gives actionable errors.

### 3. Pin Lifecycle And Metadata

- Persist CID and provider pin metadata in SQLite.
- Expose CID/gateway URLs in object and bucket responses.
- Add explicit pin status refresh for known objects.
- Attach IPFS bucket uploads to Pinata file groups for provider-side file management.
- Delete semantics: `delete_remote=true` deletes Pinata v3 files by replica locator, while shared local CID references keep the remote file.
- Keep immutable CID behavior visible to users.

Exit criteria:

- Uploading an object to an IPFS-backed route records CID metadata.
- Listing or inspecting an object shows CID and gateway URL without exposing credentials.

### 4. CDN Bucket Integration

- Add an IPFS-oriented CDN bucket workflow.
- Return stable public URL plus direct gateway URL where available.
- Support warmup/probe against selected gateway URLs.
- Verify Range GET behavior where gateways support it.
- Keep large-object limits and timeout behavior explicit.

Exit criteria:

- A user can create an IPFS-backed bucket, upload an asset and receive usable URLs.
- Probe output distinguishes Super CDN public URL, gateway URL and CID.

### 5. Website And Edge Manifest Integration

- Use IPFS for explicit durable/static asset routes, not as the default website entry layer.
- Keep the two Web hosting modes clear: `origin_assisted` is test/integration/compatibility, while `cloudflare_static` and `hybrid_edge` are the preferred production direction.
- Add IPFS route entries to edge manifests when selected.
- Include CID and gateway fallback in manifest records, and let the Worker fetch from gateway fallbacks in route mode.
- Add `ipfs-web-smoke` as the formal Web-side validation path before enabling IPFS inside broader smart routing policy tests.
- Keep Cloudflare Static as the default overseas website entry surface.
- Avoid making R2/IPFS the default website entry layer for ordinary static sites.
- Treat R2-backed Web resources as legacy compatibility. Preferred non-entry Web resources should be AList/OpenList, Cloudflare-native static assets, or IPFS/Pinata.
- Ensure static-resource failure never falls through to Go in the preferred hybrid path.

Exit criteria:

- Hybrid edge can route selected paths to IPFS gateway URLs through manifest policy.
- Entry HTML can stay on Cloudflare Static while large immutable assets can resolve through IPFS.
- `supercdnctl ipfs-web-smoke` passes against a real provider and records the gateway, CID and first-hop behavior.
- Preferred Web docs and probes make it clear whether traffic is Cloudflare entry, resource-library delivery, IPFS gateway delivery, or compatibility Go origin.

### 6. CLI And Docs

- Add or document IPFS route-profile usage.
- Add clear examples for IPFS bucket creation and upload.
- Add status/probe examples.
- Add operational guidance for gateway selection, timeouts and pin failures.
- Keep root/team auth behavior unchanged.

Exit criteria:

- A new operator can run the IPFS happy path from documentation only.
- CLI responses make CID and gateway behavior understandable.

### 7. Live Validation

- Run a small text/image object upload through IPFS.
- Run a larger asset upload if provider limits allow it.
- Verify direct gateway GET.
- Verify Super CDN public URL behavior.
- Verify probe redaction and logs.
- Verify deletion policy with a disposable object.

Exit criteria:

- One live IPFS bucket smoke is documented.
- One Web-side IPFS smoke is documented before smart routing policies are live-tested with IPFS as a candidate source. Done live with `cyberstream-ipfs-smoke` and the preferred `hybrid_edge` canary `cyberstream-ipfs-0501.qwk.ccwu.cc`.

## Non-Goals For v0.2

- Running and operating a self-hosted IPFS cluster.
- Replacing Cloudflare Static as the overseas website default.
- Replacing AList/OpenList as the domestic acceleration path.
- Expanding R2-backed website hosting as a mainstream Web path.
- Building a public SaaS UI.
- Automatic legal/content moderation workflows.

## Risks To Resolve Early

- Gateway performance and reliability can vary widely.
- Provider pin APIs and gateway behavior differ from S3-like storage.
- IPFS immutability requires clear user expectations around updates and deletes.
- Public gateway URLs may have caching, MIME or Range behavior differences.
- Very large file uploads need timeout, retry and multipart strategy decisions.

## Proposed v0.2 Definition Of Done

- `go test ./...` and `go vet ./...` pass.
- Existing v0.1 production smoke checks still pass.
- IPFS provider status check passes against real credentials.
- IPFS bucket upload returns public URL, CID and gateway URL.
- IPFS object probe succeeds against at least one gateway.
- IPFS route/profile behavior is documented in CLI reference.
- Web hosting boundary and failover rules are documented and reflected in examples.
- No root, provider or signed gateway secrets appear in logs or shared output.
